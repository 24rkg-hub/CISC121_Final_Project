# CISC121_Final_Project
**Why this algorithm fits your chosen problem and dataset**

I chose to implement Merge sort as my sorting algorithm for the Playlist Vibe Builder since the goal of the app is to sort a list of songs by a user selected key such as energy score or duration, while also showing the sorting process clearly and visually. It’s important to note that since playlists can grow to different sizes, merge sort is useful because it performs efficiently even on larger lists with a best case and worst case time complexity of O(nlog n), in comparison to Quick sort which in it’s worst case is O(n2).  It also works well for animation process because the sort naturally breaks the list into smaller parts and then merges them back together in sorted order, which makes the reordering easy for the user to follow visually. One assumption made is that this algorithm relies on every song in the dataset having the selected sorting key available in a comparable form, such as an integer energy score or duration value.

**Required preconditions and how the app checks them**

Merge sort does not require the playlist to already be sorted, which makes it a suitable choice for user-generated playlists that may start in any order. However, the app does require that the playlist contains valid song entries and that the sorting key selected by the user exists for every song inputted. For example, if the user chooses to sort by energy, each song must have an energy score between 0 and 100. If the user chooses duration, each song must have a valid duration value. In both cases, if a single song lacks a valid energy score/duration, the algorithm will not be able to successfully sort the playlist. The app can ensure these conditions are met by checking the playlist before the simulation starts. For example, if the playlist is empty, missing required data, or contains invalid values, the interface can display an error message and prevent the sort from running until the issue is fixed.

**What the user will see during the simulation**

During the simulation, the user will see the playlist displayed as a visual list of songs, with different highlights used to show what the algorithm is doing at each step. For example, when merge sort splits the playlist into smaller sections, those sections can be outlined or highlighted to show the current sublists being worked on. During the merge step, the two values being compared can be highlighted in one color to show they are actively being compared. When a song is placed into its new position in the merged list, that song can change to another color to show that it has been moved or copied into place. Once a section is fully merged and sorted, it can be highlighted to a final color to indicate that portion is complete. This helps the user understand not only the final sorted result, but also how the order is built step by step through the intermediate steps.


**Computational Thinking**
**Decomposition**

The merge sort algorithm can be broken down into several smaller steps. First, the program receives the inputs of the playlist and the user’s selected sorting key. Next, it repeatedly divides the playlist into smaller halves until each sublist contains only one song (already sorted). After that, it compares songs from the sublists beside it, based on the chosen sorting key such as the energy or duration. It then merges those songs back together in sorted order, one section at a time, until the full playlist is rebuilt. Finally, the GUI displays the completed sorted playlist to the user. Breaking the problem into these smaller decomposed actions makes the algorithm easier to design and implement.

**Pattern Recognition**

Merge sort follows a repeated pattern throughout the sorting process. It keeps splitting the playlist into smaller parts (the divide-and-conquer process), then repeatedly compares the first remaining item in each sublist during the merge stage. Whichever song has the smaller or larger target value, (depending on its intended order) is placed next into the merged result. This pattern continues until one sublist is empty after which the remaining songs from the other sublist are copied over. Since the same comparison-and-merge process happens again on smaller sections of the playlist, this shows the repeated structure in the algorithm.

**Abstraction**
It’s important to understand that not every internal detail of the merge sort algorithm needs to be displayed to the user. The most important details to display are the current sublists being merged, the songs being compared, and where songs are placed in the rebuilt playlist. These steps help the user understand the simple logic of the algorithm without providing them with irrelevant details. Less important details such as the exact variable names, the recursion call structure, or the internal indexing positions should be hidden because they are helpful for the coding algorithm but not necessary for the user experience. In the GUI, abstraction can be used by showing colour highlights, labels, and movement animations rather than displaying the raw code logic.

**Algorithm Design**
The app begins with user input through the GUI. The user can either load in or view a playlist and then selects a sorting key such as energy score or duration. Once the user presses the sort button, the program validates the playlist data ensuring there are no invalid entries and starts the merge sort process. During processing, the algorithm divides the playlist into smaller parts, compares song values, and merges them back in order while sending updates to the GUI for animation throughout the sorting process. The output is a visually sorted playlist along with step-by-step highlights showing comparisons, merges, and completed sections. In this design, the input is the playlist and sorting choice, the processing is the custom merge sort algorithm, and the output is both the final sorted playlist and the animated simulation showing how it was created.
